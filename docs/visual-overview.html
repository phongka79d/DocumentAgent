Core Workflow

Document lifecycle and question path

Upload, processing, indexing, and retrieval are split across routes and services. The upload route does not currently trigger processing automatically.

1

Upload

Validate supported file, store original in Supabase Storage, insert document metadata with status uploaded.

2

Process

Service-only path marks processing, downloads the file, parses content, chunks sections, writes chunks, builds graph rows, then marks ready or failed.

3

Index

Development endpoint embeds ready chunks through ShopAIKey and upserts vectors into Qdrant with chunk metadata payloads.

4

Retrieve

Semantic search embeds the question and queries Qdrant. Hybrid retrieval merges semantic and graph candidates.

5

Score

Hybrid scoring computes semantic, graph, keyword, metadata, position, and final scores, then applies final Top-K.

6

Answer

Planned agents verify evidence, generate cited answers, run self-check, and persist logs. Not implemented in current runtime.

Service / Module Workflow

Current services and planned agents

No runtime agent package is currently implemented. The project does define service modules now and agent plans for later milestones.

Current service workflow

Document service: validates uploads, stores original files, inserts metadata, lists/detail documents. Failure behavior includes safe HTTP errors for unsupported, empty, oversized, storage, or metadata failures.

Processing service: moves documents through processing, parsing, chunk persistence, graph build, and ready; failures mark failed with safe messages.

Retrieval services: validate questions and Top-K values, call dependencies, map results, and return empty results when no indexed chunks match.

Hybrid service: graph failures can fall back to semantic-only scoring when unavailable; missing graph rows become graph score 0.0.

Planned agent workflow

Agent 1 Retrieval: planned internal callable that validates input, calls hybrid retrieval, returns structured candidate chunks, and logs agent_1_retrieval.

Agent 2 Evidence Verification: planned verifier that accepts useful chunks, rejects weak chunks, checks contradictions, sets missing_information, and reports confidence.

Agent 3 Answer and Self-Check: planned generator that uses only verified chunks, includes file-name plus quote citations, handles insufficient evidence, and self-checks for unsupported claims.

LangGraph orchestration: planned workflow connecting the three agents, persisting runs and steps, and exposing chat, evidence, and logs APIs.

Validation and Quality Gates

Checks that protect grounded behavior

The project emphasizes backend-only execution, single-user filtering, safe dependency errors, and verified evidence before final answers.

Single-user access gate

Checks SINGLE_USER_ID ownership for documents, chunks, vectors, and future agent data. Passes when rows and searches are filtered by the configured user.

Secret boundary gate

Backend-only Supabase service role, Qdrant key, and ShopAIKey key. Failure action: do not expose provider credentials in frontend or documentation.

Upload validation gate

Allows PDF, DOCX, TXT, and CSV, with size and content-type checks where available. Unsupported or empty files fail clearly.

Processing status gate

Documents move through uploaded, processing, ready, or failed. Processing errors are mapped to safe messages.

Retrieval API gate

Empty questions and invalid Top-K values fail. Dependency errors return safe public messages. Invalid document UUIDs are validated by schemas.

Answer grounding gate

Planned Agent 3 must use verified chunks only, avoid rejected chunks and outside knowledge, include citations, and run self-check.

Example Scenario

How one question should move through the system

Illustrative example based on documented workflow. The final answer path is planned until the agent workflow is implemented.

Input event

A user uploads an employment document that states a start date of 2026-06-01, a probation duration of two months, and an official-work condition after probation.

Question: What month can official work begin?

Source-backed test scenario Answer step planned
Upload File is validated, stored in Supabase Storage, and recorded in documents.
Process Text is parsed, chunked, persisted, and graph rows are built from document-section-chunk-entity relationships.
Index Chunks are embedded through ShopAIKey and inserted into Qdrant with metadata payloads.
Retrieve Semantic and graph candidates are merged and scored, returning chunks about the start date, probation duration, and condition.
Verify Planned Agent 2 accepts chunks that directly support the answer and rejects weak or unrelated chunks.
Answer Planned Agent 3 can reason that 2026-06-01 + 2 months = 2026-08-01 and cite the verified quotes.
Risks, Limitations, and Future Work

Known boundaries to keep visible

These items come from README known gaps, the master plan non-goals, and future improvement sections.

Current runtime gaps

Upload does not automatically call processing. Processing is tested as a service but has no public trigger route. The frontend is still a placeholder.

Planned agent gaps

Public graph APIs, chat agents, LangGraph orchestration, chat APIs, evidence APIs, and agent logs APIs are not implemented yet.

Provider and rerank gaps

Live rerank provider calls are not implemented. Rerank is guarded by configuration and disabled by default behavior.

Configuration risk

Backend settings read backend/.env. A root .env exists in the workspace, but it is not loaded by backend settings by default.

MVP non-goals

No multi-user authentication, JWT login, OCR, payment system, admin dashboard, complex permissions, fine-tuning, mobile app, or browser extension.

Future improvements

OCR support, multi-user auth, graph visualization, BM25 plus vector search, advanced reranking, streaming answers, document comparison, source previews, summaries, and export.
